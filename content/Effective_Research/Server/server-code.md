# 复现与迁移

学会使用 `docker` 或 `podman` 等容器，我一般使用 `docker`（目前大部分都还是使用的 `docker`）

## 复现

复现代码分为：

1. 拉取开源代码进行实验的复现
2. 阅读论文手写代码复现，并与实验进行对比

### 实验复现

> 当前 `github` 必须要在使用的机器上认证，才能够拉取代码

工作流程如下：

1. `git clone <url>` 拉取代码
2. 阅读 `README` 或 `INSTALL` 等文档，一般都会有写如何复现
3. 如果文件目录中没有看见 `Dockerfile`，请自己手写一个 `Dockerfile` 进行复现，否则使用他的

如何手写 `Dockerfile`，请看教程 [Docker实践](https://yeasy.gitbook.io/docker_practice/)

手写时请注意，需要你确定 `README` 中所提到的包并在 `docker` 中安装。

> 请不要在本地进行实验复现，随着复现的实验越来越多，你的本地包环境可能会变得十分杂乱，到最后产生版本不兼任等问题，会浪费很多时间来解决类似这种环境问题。

### 开发复现

在这种情况下，我并不推荐你使用 `docker` 环境，可以直接在本地进行开发，使用 `docker` 环境进行测试

本地的系统最好也是 `Linux`，这样就可以省去迁移的步骤。如果你使用 `Windows` ，也那么可以使用 `WSL` 进行开发。

> `WSL` 最好选择与你将要迁移过去的系统一样的发行版（版本可以不一样）

## 迁移

当你的代码完成后，需要迁移到服务器上跑实验数据，请写 `Dockerfile` 开一个自己的 `docker` 来跑数据，不要污染服务器的包环境（并且你可能也并没有 `sudo` 权限）

因此，本地验证完正确性后，请将代码的运行环境迁移到服务器的 `docker` 上，不要污染服务器的包环境。
